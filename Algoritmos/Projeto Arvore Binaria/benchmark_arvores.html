⚠ RESULTADO CONTRAINTUITIVO: O índice auxiliar foi MAIS LENTO em 3 de 4 buscas não-chave. Por quê?

O índice usa: dict.get(campo, valor) → lista de k IDs → loop: self.buscar(sid) para cada ID → k × O(log n)
Traversal O(n) usa: visita cada nó uma vez → n × O(1)

Para aprovados (k=4.368): índice faz ≈ 4.368 × log(10.507) ≈ 58.400 operações. Traversal faz ≈ 10.507 operações.
O índice só vence quando k << n / log(n) ≈ 784 registros. Nenhuma das buscas testadas tem resultado tão pequeno.
Quando o índice auxiliar VALE a pena?
Limiar de speedup — índice vs. traversal O(n)
// O índice vence quando k (resultados encontrados) < n / log₂(n) ≈ 784
Regra prática: O índice é mais rápido quando o número de resultados k satisfaz k × log₂(n) < n.
Para n=10.507: limiar ≈ 10.507 / log₂(10.507) ≈ 784 resultados.

• Aprovados (k=4.368) → 58.400 ops vs 10.507 → traversal 5,6× mais rápido ✓ confirmado no benchmark
• Categoria A (k=28) → 375 ops vs 10.507 → índice 28× mais rápido ✓ índice valeria aqui
• Um campo com k=100 → 1.340 ops vs 10.507 → índice 7,8× mais rápido ✓

Tempo de inserção — 10.507 registros (ms)
// ABB+Idx: inserção BST + construção do índice (percurso O(n) após inserção)
A AVL é 2,7× mais lenta que a ABB na inserção (109,5 ms vs 40,5 ms). A cada inserção, a AVL executa: inserção padrão + atualização de altura + verificação do fator de balanceamento + rotação quando |FB| > 1. Com 10.507 nós, esse overhead acumulou ~69 ms extras.

O ABB+Idx completou em apenas 39,4 ms pois a construção do índice foi quase instantânea (apenas percorreu a árvore já existente). O AVL+Idx levou 9,97 ms a mais para construir o índice sobre a AVL.
